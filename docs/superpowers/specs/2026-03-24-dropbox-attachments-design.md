# Dropbox Image Support & Generalized Attachment Extraction

**Date**: 2026-03-24
**Status**: Approved

## Problem

og-cal extracts images from Google Drive URLs in event descriptions but doesn't handle Dropbox share links. A live event on nobigbendwall.org has three Dropbox image URLs that render as broken images. Additionally, non-image file links (PDFs, docs, etc.) from any host are ignored entirely — the og-cal schema has an `attachments` field that's never populated.

## Solution

Two changes:

1. **Extend image extraction** to recognize Dropbox share URLs and normalize them to direct-servable image URLs.
2. **Add generalized attachment extraction** that detects file URLs (PDF, doc, spreadsheet, zip, etc.) from any host and populates the `attachments` schema field. Render attachments in the detail view.

## Design

### Image extraction (images.js)

**New Dropbox patterns** alongside existing Drive support:
- `dropbox.com/scl/fi/` (current format)
- `dropbox.com/s/` (legacy format)
- `dl.dropboxusercontent.com` (direct download subdomain)

**Normalization**: Replace `dl=0` with `raw=1` (or append `?raw=1` if no query) to get a direct-servable URL.

**Image vs. non-image filtering**:
- Dropbox URLs with image extensions (`.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`) → extracted as images
- Dropbox URLs with known non-image extensions (`.pdf`, `.docx`, `.zip`, etc.) → skipped, picked up by attachment extraction
- Dropbox URLs with no recognizable extension → optimistic image treatment (try rendering, broken-image CSS hides failures)

`normalizeImageUrl()` is extended to handle both Drive and Dropbox URLs.

### Attachment extraction (new: attachments.js)

A new `extractAttachments()` function, **host-agnostic**:

**Detection**: Scans description for URLs whose path contains a known file extension:
- Documents: `.pdf`, `.doc`, `.docx`
- Spreadsheets: `.xls`, `.xlsx`, `.csv`
- Presentations: `.ppt`, `.pptx`
- Archives: `.zip`
- Text: `.txt`

Works regardless of host — Dropbox, Drive, S3, any raw URL.

**Labeling**: Labels generated by broad file type category:
- `"Download PDF"`, `"Download Document"`, `"Download Spreadsheet"`, `"Download Presentation"`, `"Download Archive"`, `"Download File"`

**URL normalization for known hosts**:
- Dropbox: `dl=0` → `raw=1` for direct download
- Google Drive: `/uc?export=download&id={fileId}` variant for non-image files
- All other hosts: URL passed through as-is

**Schema** (populates existing `attachments` field):
```json
{
  "label": "Download PDF",
  "url": "https://www.dropbox.com/scl/fi/.../flyer.pdf?rlkey=...&raw=1",
  "type": "pdf"
}
```

**Stripping**: Extracted attachment URLs removed from description text, same `stripUrl()` pattern as images and links.

**Execution order** in `data.js` event enrichment:
1. `extractImage()` — images (including Dropbox/Drive image URLs)
2. `extractLinks()` — platform links (Eventbrite, Instagram, etc.)
3. `extractAttachments()` — remaining file URLs as attachments

Each function receives the already-cleaned description from the previous step, preventing double-extraction.

### Detail view UI (views/detail.js)

- New section below description, above existing platform link buttons
- Each attachment rendered as a button/link, same visual style as platform links
- Text prefix indicating file type (e.g., "PDF", "DOC") — no icon library
- Opens in new tab (`target="_blank"`)
- Section omitted when no attachments (same pattern as links)

### Testing

- `test/images.test.js` — Dropbox image URL extraction and normalization (all three URL formats, extension filtering, optimistic extensionless handling, dedup)
- `test/attachments.test.js` — new file: PDF extraction, docx, zip, mixed hosts, Dropbox normalization, Drive normalization, deduplication with already-extracted images/links, labeling, edge cases

## Out of Scope

- PDF thumbnail previews or inline rendering
- File type icons (icon library)
- Download progress indicators
- OneDrive/Box/S3-specific URL normalization (URLs pass through as-is, which works fine)
